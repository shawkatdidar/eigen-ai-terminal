## Today I Learned

### Sparse MoE: Why "398B Parameters" and "13B Active" Are Not Contradictory

**Why this came up:** Three major signals today involve MoE (Mixture of Experts): Gemma 4's 26B MoE with 4B active, Arcee Trinity's 398B with 13B active, and Alibaba's Qwen3.6-Plus architecture.

You've seen MoE before — you know only a fraction of parameters activate per input. Today's signals make the economic implication concrete: Arcee Trinity has 398B total parameters (meaning 398 billion learnable values spread across many expert sub-networks) but only activates 13B per input token. It ranks #2 on PinchBench behind only Claude Opus 4.6 — which uses all of its parameters for every token. Trinity costs $0.90/M output tokens. Opus costs ~$22.50/M. That's a 96% cost reduction for near-equivalent agent capability.

The mechanism: a router network (itself a small neural network) examines each input token and decides which 2-4 expert sub-networks (out of dozens) to activate. The remaining experts sit idle for that token. This means the model has 398B parameters of *knowledge capacity* (what patterns it can recognize) but only 13B parameters of *compute cost* (what it actually calculates per token). Knowledge is stored wide; computation stays narrow.

### Quick Glossary
| Term | Meaning |
|------|---------|
| PLE (Per-Layer Embeddings) | Each transformer layer gets its own embedding projection rather than sharing one global embedding, allowing layers to specialize |
| Shared KV Cache | Multiple attention layers reuse the same key-value cache entries, reducing memory during inference |
| PinchBench | Agent capability benchmark measuring multi-step task completion by AI models |
| Terminal-Bench | Benchmark for coding agent performance in terminal/CLI environments |
| SWE-bench | Software engineering benchmark testing ability to resolve real GitHub issues |
| FLEURS | Multilingual speech benchmark covering 100+ languages for evaluating speech recognition |
