## Today I Learned

### Benchmark Saturation vs. Generalization

**Why this came up:** Stanford AI Index 2026 — SWE-bench Verified went from 60% to ~100% in one year, while the same models that win International Mathematical Olympiad gold correctly read analog clocks only 50.1% of the time.

**The mechanism.** A benchmark measures a specific capability as that capability appears in the benchmark's task distribution. SWE-bench Verified is a curated set of GitHub issues where (1) the problem has a known correct patch, (2) the tests are deterministic, (3) the repo setup is reproducible, (4) the issue is stated in a specific form. Training on SWE-bench-like data — or training with SWE-bench-like verification loops in the reward signal — produces models that close distance to 100% on that distribution. What it does NOT produce is models that generalize to "read the clock on the wall," "count the items in this pantry photo," or any other task whose training signal wasn't explicitly in the optimization loop.

**Why "saturation" doesn't mean "solved."** When a benchmark moves from 60% to 100% in 12 months, the first-order reading is "the capability it measures is solved." That reading is often wrong. The second-order reading is: "the distribution inside that benchmark is learned; the capability the benchmark was intended to proxy may or may not be." SWE-bench was intended to proxy "general software-engineering competence." It has been saturated by systems that are simultaneously unable to reliably read an analog clock — a visual perception task a child performs without thinking.

**The judgment framework.** When someone says "model X scores Y% on benchmark Z," the three questions that separate a real capability claim from a benchmark-hacking claim:

1. **What tasks OUTSIDE the benchmark's distribution does X do?** If you can't produce examples, the benchmark saturation is probably narrow.
2. **What does X get wrong that a 10-year-old gets right?** The answer reveals the shape of the actual capability gap — these are the tasks that haven't been gradient-descended into the weights yet.
3. **Is the benchmark itself still a discriminator?** If 5 frontier models all score 95%+, the benchmark has expired. Noise becomes larger than signal above ~90% on any fixed benchmark.

**Applied to today's signals:** Claude Opus 4.7 at SWE-bench Pro 64.3%, Qwen3.6-35B-A3B matching Opus 4.6 on LiveCodeBench — these are real but the benchmarks themselves are nearing their discrimination ceiling. The next set of frontier comparisons (late 2026) will need benchmarks that haven't been saturated, which likely means benchmarks that touch visual perception, physical reasoning, or multi-hour task coherence — the exact spots where current frontier models fail.

### Protocol-Level Trust vs. Vendor-Level Trust

**Why this came up:** OX Security disclosed a remote code execution vulnerability in MCP — Anthropic's Model Context Protocol — affecting 150M+ downloads across all major MCP implementations.

**The mechanism.** A protocol is a shared contract between parties. MCP says: "A server exposes tools with schemas; a client calls the tools; responses come back in a structured form." A flaw at the protocol level — where the contract itself is unsafe — is different from a flaw at the vendor level, where a specific implementation has a bug. Vendor bugs get patched by one vendor; protocol flaws require every implementer to update in coordinated fashion, AND the protocol spec itself to update, AND backwards compatibility shims to be resolved.

**Why MCP is particularly exposed.** MCP went from Anthropic-internal → Anthropic-published spec → adopted by OpenAI (sunsetting their proprietary Assistants API) → Slack/Salesforce → Mistral → GitHub → Microsoft → dozens of enterprise tools — in roughly 12 months. This is faster-than-normal protocol adoption because the ecosystem was desperate for a standard. Fast adoption concentrated the surface area. A single RCE at the protocol trust-model level is now an industry-wide problem.

**The judgment framework.** When evaluating the safety of a system that composes multiple vendors via a protocol, ask:

1. **Who controls the spec?** (Anthropic, in MCP's case — which means one vendor sets the upgrade path for everyone else.)
2. **What is the protocol's trust model?** (MCP was built assuming MCP servers are trusted by the client — which turns out to be the wrong default when MCP servers can be third-party community-contributed.)
3. **What happens when a vulnerability is found at the spec level, not just a vendor level?** (For MCP, the answer right now is "we're about to find out.")

**Applied to today's signal:** The OX Security disclosure is not a Claude Code bug, a Cursor bug, or a Slack bug. It is a protocol-level trust failure. The fix requires Anthropic to update the spec, all implementers to adopt the update, and all MCP server developers to ship compliant responses. That coordination — across competitors who adopted MCP precisely because they needed a non-Anthropic-controlled interop story — is the real test of MCP's governance model. The outcome will determine whether MCP becomes the enterprise default (like HTTP) or gets replaced (like SOAP).
